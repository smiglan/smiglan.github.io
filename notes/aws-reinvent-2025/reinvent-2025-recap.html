AWS re:Invent 2025: The Year Infrastructure Caught Up to Agents

Published notes and synthesis from AWS re:Invent 2025.

AWS re:Invent 2025: The Year Infrastructure Caught Up to Agents¶

I just got back from my first re:Invent. Over 60,000 people descending on Las Vegas to watch AWS announce… infrastructure for AI agents. Which sounds boring until you realize: boring infrastructure is exactly what production systems need.

The conference had the usual spectacle—giant keynotes, endless expo halls, that particular brand of Vegas-meets-tech-conference surrealism. But the useful part wasn’t the announcements themselves. It was getting 20 minutes with the people actually building these services, understanding what problems they’re actually solving, and figuring out what that means for what we should build next.

Here’s what stuck with me.

The thing nobody says out loud about agents¶

Most agent demos die the same way: they work great in the demo, then you try to put them in production and discover that 80% of your time is spent on problems that have nothing to do with the model.

The model isn’t calling tools correctly. Your retrieval system is slow and expensive. You can’t audit what the agent decided or why. Your authentication story is “we’ll figure it out later.” You’re burning tokens on redundant context. The whole thing falls over when someone changes an API.

This year felt different because AWS is finally shipping the boring parts. Not “better models” (though they announced those too). The boring parts. The infrastructure.

Agent Core: stateful compute that doesn’t bankrupt you¶

Marc Brooker (AWS’s Vice President and Distinguished Engineer for Agentic AI) did a deep dive on Agent Core, and one detail jumped out: the cost model bills based on actual CPU consumption and memory usage per second. During I/O wait periods (when the agent is waiting for LLM responses, tool calls, or API responses), if no CPU is being consumed, you’re only paying for memory.

This is massive for long-running agent workflows. Most agents spend 30-70% of their time in I/O wait: waiting for a tool to return, waiting for a human approval, waiting for some external API. Traditional compute services charge for pre-allocated resources regardless of whether you’re using them. Agent Core’s consumption-based pricing means idle CPU time during waits is essentially free.

Agent Core is built on Firecracker (same tech that powers Lambda), but redesigned specifically for agents: - Extended runtime windows for long-running workflows - Stateful memory that persists across tool calls - No compute billing during waiting periods

The other interesting bit: they’re adding a policy framework that uses neuro-symbolic AI internally. That’s a fancy way of saying “deterministic constraints + LLM reasoning.” You can now specify what an agent is allowed to do with mathematical certainty, not just prompt engineering and crossed fingers.

For regulated industries (healthcare, finance, government), this is the difference between “interesting prototype” and “we can actually deploy this.”

The retrieval system you didn’t know you needed¶

If you’ve built anything with RAG, you know: retrieval is where things get expensive and slow.

AWS announced two things that form a natural tiering strategy:

OpenSearch Serverless with GPU acceleration¶

  • ~100ms latency for vector search
  • Up to 10× performance improvement
  • Auto-optimization for indexing

The catch: GPU acceleration only kicks in when your data segments exceed 50MB. For a lot of workloads, you’ll never hit that threshold. AWS was refreshingly honest about this in the sessions. If you need ultra-low latency at smaller scale, use OpenSearch Managed instead of Serverless.

S3 Vectors (now GA)¶

  • Sub-second latency (<1s)
  • Up to 90% cheaper than traditional vector databases
  • Perfect for “cold” vectors that don’t need sub-100ms retrieval

The smart architecture is obvious once you see it: hot vectors in OpenSearch, cold vectors in S3. This is the same pattern we’ve used for databases forever. We’re just finally applying it to embeddings.

One architect I talked to is moving everything older than a week to S3 and keeping only recent/frequently-accessed vectors hot. Cost dropped by 70%, latency barely changed for their use case.

Nova Act: when the API doesn’t exist¶

The most surprising session was on Nova Act and Browser Use. This is AWS’s answer to “what if the system you need to automate doesn’t have an API?”

Here’s how it works: 1. Agent runs through a workflow in a sandboxed browser (agentic mode, ~45 seconds) 2. Records a trajectory of actions 3. Replays it deterministically (~15 seconds) 4. If the UI changes, automatically falls back to agentic reasoning and creates a new trajectory

Full observability: screenshots, step logs, reasoning traces.

The use cases are more real than I expected: - 1Password uses it to test login flows across thousands of sites (because every site’s login flow is slightly different and breaks in its own special way) - Blue Origin uses it for UI testing

The insight: don’t replace Selenium/Playwright. Augment them. Run deterministic automation where you can, inject LLM reasoning only where the UI is unpredictable or changing.

Bedrock Data Automation: expensive but powerful¶

Bedrock Knowledge Bases now support multimodal embeddings: text, audio, video, images. Unified ingestion pipeline. Sounds great.

The reality check from the sessions: this is prohibitively expensive for high-volume ingestion. Embedding costs scale linearly with: - Frame sampling for video - Page-level image embeddings
- Audio chunking

If you’re processing thousands of multimedia files per day, the bill will hurt. But if you have a smaller, curated knowledge base (FAQs, compliance docs, training materials), the convenience is worth it.

Know your scale before you commit.

Kiro: momentum in AWS-native development¶

Kiro wasn’t new at re:Invent (it was announced earlier), but the number of sessions dedicated to it and the momentum behind it were notable. I’ve been using it with Claude Opus 4.5 for the past few weeks, and it’s hit that inflection point where it actually works on real, hard problems.

What makes Kiro valuable is its deep integration with AWS infrastructure. It has native access to your AWS resources: CodeCommit, CloudFormation, Lambda functions, S3, DynamoDB, and the rest of your stack. This isn’t just about code completion. Kiro can read your infrastructure state, understand your deployment patterns, and generate code that fits how your systems actually work.

The practical difference: when you ask it to add a new API endpoint, it knows your existing API Gateway configuration, follows your Lambda naming conventions, updates the right IAM policies, and modifies your CloudFormation templates. It’s working with your actual infrastructure, not guessing.

Tools like Claude Code or Cursor might have similar general coding capabilities, and they’re excellent for what they do. But if you’re already building on AWS, Kiro’s native integrations mean you don’t spend time explaining your infrastructure to the agent. It already knows.

AWS is using Kiro internally across the company, which gives me more confidence than the usual “we built this for customers” pitch. Matt Garman shared one example: a 30-developer project estimated at 18 months was completed in 76 days with 6 developers.

The spec-driven approach (Requirements, Design, Tasks) adds process, but it’s how you’d actually work on production systems. For teams deep in the AWS ecosystem, the infrastructure awareness makes Kiro worth evaluating.

Key takeaways for building production agents¶

Based on what I learned across sessions and conversations with AWS engineers, here are the architectural decisions I’m prioritizing:

Treat retrieval as infrastructure, not a feature. Stop thinking of vector search as “just embed and query.” This means benchmarking p95 latency like you would for a database, planning hot/cold storage tiers from day one, budgeting for index rebuilds explicitly, and monitoring tail latency and timeouts. One architect I met cut costs by 70% by moving vectors older than a week to S3 while keeping only recent/hot vectors in OpenSearch.

Schema-validate everything. Use structured outputs (Bedrock Responses API standardizes this). Reject or repair invalid tool calls explicitly rather than silently trying to recover. Most agent failures in production aren’t intelligence failures, they’re interface failures. Make outputs machine-checkable.

Put tools behind a protocol boundary. Use MCP or a similar standard to centralize authentication, rate limiting, and audit logging at the gateway layer. This turns tool integration from “agent hack” into “platform capability” and makes the system auditable and governable.

Match your cost model to your workflow. If your agent spends significant time in I/O wait (most do), Agent Core’s consumption-based pricing can reduce costs substantially since you’re only paying for memory when no CPU is consumed. If you’re doing continuous compute, Lambda might be cheaper. Run the numbers for your specific use case.

Consider spec-driven development for production codebases. The Kiro approach of defining requirements and design before writing code adds upfront process, but it produces maintainable, documented software that actually reflects your team’s standards. For complex multi-repo work or regulated environments, this structure is what makes agentic coding production-ready rather than just fast prototyping.

The thing I didn’t expect: ROI is already real¶

Multiple organizations across different industries reported 3-4× ROI on early agent deployments. Not projections. Actual, measured returns from systems in production.

The catch: every successful deployment was built on partnerships and iteration, not just “we bought a model and shipped it.” This stuff still requires real engineering, domain expertise, and continuous improvement.

But the economics are starting to work.

Measuring what matters¶

If there’s one thing I’m stealing from these sessions, it’s the metrics. Stop measuring “vibes.” Measure:

  • Tool call validity rate: % of calls passing schema validation
  • Tool call success rate: % succeeding on first try (no retries)
  • Evidence coverage: % of answers citing retrieved documents
  • Retrieval p95 latency: tail latency kills user experience
  • Cost per successful task: dollars per completed workflow

These turn “it feels better” into “it is 23% better and here’s the chart.”

Why you should go¶

The announcements are public. You can read the blog posts. But here’s what you can’t get remotely:

Direct access to the people building these services. Twenty minutes with a principal engineer answering “okay but what about X?” is worth more than hours of documentation. You get to understand not just what shipped, but what problems they’re actually solving and what’s coming next.

Pattern recognition across industries. Hearing what five different companies are doing with the same primitive surfaces patterns you wouldn’t discover alone. The problem you’re stuck on might already be solved in a completely different domain.

Serendipitous connections. The person you chat with at a workshop might be the one who unblocks you three months from now. These relationships compound over time.

Real feedback loops. When you tell an AWS engineer “this doesn’t work for our use case,” they often take notes and follow up. Your production pain points can influence the roadmap.

The scale is massive (over 60,000 attendees), the logistics require planning (venues span miles of the Las Vegas Strip, so pick your hotel strategically based on your session clusters), and you’ll absolutely need comfortable shoes (expect 20,000+ steps per day).

But if you’re building anything in this space, the access and pattern recognition are worth it.
